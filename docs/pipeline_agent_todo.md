## Task 9: Ledger Guard False Positive on Code Comments 🟡 P2

**Status:** ✅ Implemented 2026-05-03

This prevents false positives on e.g. `// ### [DebugLog]` and `-- ### [SomeHeader]` inside C++ or Lua code fences.

## Task 10: Insanity Detector Bypass (Alternating Loop) 🟡 P2

**Status:** ✅ Already Handled (Existing Code)

**Files:** `pipeline.py` (Phase 6 review-fix loop)
**Discovered:** 2026-05-02 Audit

**Issue:** The circuit breaker uses `hashlib.md5()` to compare only the **last** output against the **current** output. An alternating loop (A → B → A → B) will never trigger a repeat match, bypassing the circuit breaker entirely.

**Verdict:** Upon inspection, the existing code (now at L4121) already uses a `set()` (not just a single hash):

```python
seen_code_hashes = set()
...
code_hash = hashlib.md5(fix_output.encode("utf-8")).hexdigest()
if code_hash in seen_code_hashes:
    # Circuit breaker: trip BLOCKED
    ...
seen_code_hashes.add(code_hash)
```

A `set` of ALL historical hashes catches alternating loops (A-B-A-B) because A will be found in the set when it repeats. **No code change needed.** This task is marked as Done.

## Task 11: Token Budget Ratio Overflow 🟢 P3

**Status:** ✅ Implemented 2026-05-03

**Files:** `pipeline.py` (`TokenBudget.estimate_tokens()`)
**Discovered:** 2026-05-02 Audit

**What was missing:** The `estimate_tokens()` method used a flat 3.0 char/token ratio for all content. Dense data (base64, hex arrays, Unicode) causes actual token count to vastly exceed estimates, leading to Ollama truncating the JSON request mid-stream (see `docs/pipeline_agent_todo.md` for audit record).

**Implementation:**
- Modified `TokenBudget.estimate_tokens()` to add a **density-aware safety margin**
- Sampled the first 2000 characters of input text
- If alphanumeric density > 60%, uses a conservative 1.5 char/token ratio (`len(text) // 2`)
- Otherwise defaults to standard 3.0 char/token ratio (`len(text) // 3`)
- Updated the method's docstring to document both code paths and explain the `Token Budget Ratio Overflow` failure mode

**Updated method (pipeline.py L104):**
```python
def estimate_tokens(self, text: str) -> int:
    """Rough token estimate with density-aware safety margin.
    For normal code/text, uses 1 token per 3 chars (standard heuristic).
    For dense data (base64, hex arrays, Unicode) where alphanumeric
    density > 60%, uses a conservative 1.5 char/token ratio to prevent
    Ollama truncating the JSON request mid-stream (Token Budget Ratio Overflow).
    """
    sample = text[:2000]
    if sample:
        alpha_count = sum(1 for c in sample if c.isalnum())
        density = alpha_count / len(sample)
        if density > 0.60:
            return len(text) // 2  # 1.5 char/token conservative
    return len(text) // 3  # default: 1 token per 3 chars
```

## Task 14: Locked File Lifecycle — Edge Case Investigation 🟡 P2

**Status:** ✅ Implemented 2026-05-03
**Files:** `pipeline.py` (L672, L675, L2691), `pipeline_stream_server.py` (L117-148, L286-292)
**Created:** 2026-05-03

**Context:** Tasks 4 and 5 introduced `threading.Lock` guards (SESSION_TIMELINE_LOCK, LEDGER_WRITE_LOCK, GLOBAL_MESH_LOCK) for safe concurrent file I/O between the SSE streaming server's background thread and the main pipeline thread. This task audits the lifecycle of these locked file operations under abnormal conditions.

### Implementation

#### 1. Lock Timeout Added to SESSION_TIMELINE_LOCK and LEDGER_WRITE_LOCK ✅

Both locks now use `_timed_lock()` helper instead of raw `threading.Lock`:
```python
def _timed_lock(lock_obj, timeout=30):
    """Acquire a threading.Lock with timeout. Returns True if acquired, False if timed out."""
    return lock_obj.acquire(timeout=timeout)
```

Usage pattern:
```python
if not _timed_lock(SESSION_TIMELINE_LOCK):
    print("[LockTimeout] SESSION_TIMELINE_LOCK timed out after 30s — skipping timeline write")
    return
try:
    # ...locked operations...
finally:
    SESSION_TIMELINE_LOCK.release()
```

This prevents indefinite deadlocks if the filesystem hangs during `read_text()` or `write_text()` inside the lock.

#### 2. Atomic File Writes in `log_to_session_timeline()` ✅

The read-prepend-write pattern now uses write-to-tmp + `os.rename()` for atomicity:
```python
# Write to temporary file, then atomically rename
tmp_path = session_timeline_path.with_suffix(".tmp")
tmp_path.write_text(new_content, encoding="utf-8")
os.rename(str(tmp_path), str(session_timeline_path))
```

This prevents partial writes from producing a truncated session timeline if the process crashes during `write_text()`.

#### 3. `mkdir()` Moved Inside Lock in `_append_to_ledger()` ✅

`ledger_path.parent.mkdir(parents=True, exist_ok=True)` is now called inside the `LEDGER_WRITE_LOCK` block, eliminating the TOCTOU race condition.

#### 4. `GLOBAL_MESH_LOCK` Dead Code — Note Added ✅

`GLOBAL_MESH_LOCK` is retained as a reserved variable for future mesh queue concurrent access. Added a comment explaining it is intentionally unused until mesh work queue threading is implemented.

#### 5. `thread.join()` Blocking Note Added ✅

Added a detailed block comment in `pipeline_stream_server.py` at the `thread.join()` call site (pipeline_stream_server.py L286-292) documenting the blocking behavior and its implications for future `ThreadingHTTPServer` upgrades.

### Observed Edge Cases

1. **No timeout on locks** — ✅ **Fixed**: `_timed_lock()` with 30s timeout
2. **Process crash during read-prepend-write** — ✅ **Fixed**: Atomic write-to-tmp + os.rename()
3. **mkdir() outside lock in _append_to_ledger()** — ✅ **Fixed**: Moved mkdir inside LEDGER_WRITE_LOCK
4. **GLOBAL_MESH_LOCK dead code** — ✅ **Documented**: Reserved for future use with comment
5. **Stream server thread.join() blocks HTTP handler** — ✅ **Documented**: Block comment explaining single-threaded HTTPServer behavior

### Investigation Checklist

## Task 18: Pre-Truncation Hash Validation ✅

**Status:** ✅ Implemented 2026-05-03

**Files:** `pipeline.py` (Phase 6 review-fix loop, new `_normalize_fix_fingerprint()`)

**Issue:** The Insanity Detector (Phase 6 circuit breaker) computed `hashlib.md5(fix_output)` on the LLM's output text. Since `call_ollama()` internally calls `_block_aware_collapse` on the INPUT context, identical fix requests could produce different LLM outputs (due to different functions being collapsed in different cycles), causing the hash-based loop detection to silently miss infinite loops.

**Implementation (Approach #1: Pre-truncation fingerprint hashing):**

1. **Added `_normalize_fix_fingerprint(fix_input)` helper** (L675) — a deterministic normalization function that:
   - Normalizes cycle numbers (`"Cycle 2"` → `"Cycle N"`) to eliminate timestamp drift
   - Strips `[... body collapsed ...]` / `[... truncated ...]` artifacts (defense-in-depth)
   - Collapses multiple blank lines to one (whitespace normalization)
   - Returns a canonical fingerprint of what the pipeline *intends* to ask the LLM

2. **Moved hash computation to BEFORE `call_ollama()`** in the fix loop (L675):
   - Replaced `code_hash = hashlib.md5(fix_output.encode(...)).hexdigest()` with:
     ```python
     finger_hash = hashlib.md5(
         _normalize_fix_fingerprint(fix_input).encode("utf-8")
     ).hexdigest()
     ```
   - The hash is now computed on `fix_input` (immutable — identical fix requests produce identical hashes regardless of context budget pressure)

3. **Updated the Insanity Detector docstring** (L675-2766) to document the pre-truncation fingerprint approach and reference `_normalize_fix_fingerprint()`.

**Why this is better:** The fingerprint is computed on immutable data (what we're asking), before any context truncation happens inside `call_ollama()`. This means `_block_aware_collapse` can corrupt the context window however it likes — the hash will always produce the same result for the same fix request, catching genuine infinite loops that the old post-output hashing missed.

3. In `apply_proposals()` — before `shutil.copy2()`, reads the proposal file from disk as bytes, computes its MD5 hash, and compares against the stored `content_hash` from manifest. On match → proceeds with copy. On mismatch → returns `(rel_path, False, "Hash mismatch — possible silent file corruption")` and skips the copy. Legacy entries (pre-hash) get a warning but still succeed.
4. The `hashlib.md5(disk_bytes)` call uses `read_bytes()` (binary mode) to handle any encoding issues identically to the utf-8 hash computed at save time.

**Why this works:** The hash is computed at write time in `save_proposal()` and verified at apply time on the exact file on disk — catching any silent corruption that occurred between save and apply. MD5 is sufficient for corruption detection (not a security boundary).

## Task 20: Live Stream Keep-Alive Heartbeat

**Status:** ✅ Implemented 2026-05-03

**Files:** `pipeline_stream_server.py`

**Issue:** During lengthy 3-5 second model-loading phases, the VS Code client proxy would timeout and abort the connection with WinError 10053. No data was being sent over the SSE socket during these quiet periods.

**Implementation:**

Added a heartbeat daemon thread that runs in parallel with the pipeline, sending SSE ping packets at 1-second intervals:

1. **`_send_ping(wfile)`** — Sends an SSE `event: ping\ndata: {}\n\n` packet to keep the socket alive. This is SSE-spec compliant and VS Code's EventSource parser ignores unknown event types gracefully.

2. **`_heartbeat_loop(wfile, stop_event, interval=1.0)`** — Daemon thread function that loops, calling `_send_ping()` every `interval` seconds until `stop_event` is set. Handles client disconnection gracefully by breaking out on `BrokenPipeError`/`ConnectionResetError`.

3. **Modified `_run_pipeline_in_thread()`** — Creates a `threading.Event` stop signal, starts the heartbeat daemon, runs the pipeline, then signals the heartbeat to stop and joins it (with 2s timeout) before sending final termination signals.

The heartbeat runs during the model-loading gap AND any other quiet periods where no tokens are being streamed, keeping the socket alive throughout the entire pipeline execution.
